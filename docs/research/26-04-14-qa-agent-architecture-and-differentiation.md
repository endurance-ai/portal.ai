# Q&A 에이전트 아키텍처 & 차별화 전략

> - 작성일: 2026-04-14
> - 목적: portal.ai의 "체계적 Q&A 기반 패션 추천"이 왜 다른지, 어떻게 작동하고, 어디로 가는지를 한 문서로 정리

---

## 1. Core Thesis

> Daydream은 유저에게 취향을 언어로 쓰라고 요구한다. 우리는 답할 수 있는 질문을 순서대로 던져, 유저가 이미 가진 취향을 한 단계씩 정리해준다.

---

## 2. 왜 자유 프롬프트가 안 되는가

### 유저는 취향을 말로 쓸 수 없다

55명 인터뷰 중 텍스트 검색으로 탐색을 시작한 유저: **0명**. 전원이 인스타 릴스, 유튜브 하울, 핀터레스트 등 비주얼 브라우징에서 시작했다.

> **허자연**: "추구미는 있는데 용어도 잘 모르고 진짜 딱 맞는 용어가 있을텐데 그걸 모른다. 프롬프트 쓰기도 어려울 수 있다."
>
> **여지혜** (무신사 업계 종사자): "자기가 지금 찾고 있는 네이밍 자체를 모르는 사람들이 많음. 뭔가를 가지고 싶은데 그게 뭔지를 정확하게 모름."

Daydream은 "revenge dress for a wedding in Paris"처럼 유저가 자연어로 복잡한 쿼리를 작성할 수 있다는 전제 위에 설계되어 있다. **한국 니치 유저에게 그 전제는 성립하지 않는다.**

### 자유 프롬프트의 구조적 한계

**송진우** — Daydream 실사용자이자 portal.ai 프로토타입 테스터:

> "재검색 시 내가 처음에 검색한 범주를 안 벗어나는 게 중요할 것 같다고 느낌"
>
> "너무 다양하게 추천해서 '괜찮은데 싶다가도 이게 필요한 게 아니라서 망설여짐'"
>
> "오히려 자유도를 낮췄으면 좋겠다"

자유 프롬프트는 해석의 부담을 AI에게 넘기고, 그 결과 **대화할수록 원래 원하던 것에서 멀어진다.** 이것이 Daydream의 구조적 실패 모드다.

### GABI에서 얻은 착안점

올웨이즈의 가전비서(GABI)는 유저에게 "어떤 냉장고를 원하세요?"라고 묻지 않는다. 대신 "가족이 몇 명이세요?", "얼마나 자주 요리하세요?" 같은 **답할 수 있는 질문**을 하고, 그 답으로 원하는 스펙을 AI가 추론한다.

이 패턴이 **송진우의 요청("자유도를 낮춰라")과 정확히 일치**한다는 발견이 전환점이었다.

## 3. 우리의 접근: 6-Step Q&A

레퍼런스 하나에서 출발해, 단계마다 유저에게 "답할 수 있는 질문"을 던지며 검색 공간을 좁혀간다.

| Step | 유저에게 묻는 것 | 시스템이 얻는 것 |
|------|-----------------|-----------------|
| **1. Reference** | "어떤 스타일을 찾고 계세요?" (이미지/텍스트) | 아이템 분해 + 속성 추출 (category, fit, fabric, color, season, pattern) |
| **2. Confirm** | "이렇게 찾아볼게요. 맞나요?" + 수정 가능 | AI 분석 검증 + 유저 수정 → 정확한 속성 확정 |
| **3. Hold** | "꼭 지켜야 할 부분은?" (max 3) | hard filter 조건 확정 → 이 속성은 절대 이탈 안 함 |
| **4. Conditions** | "얼마나 비슷하게? 예산은?" | 유사도 수준 (tight/medium/loose) + 가격 범위 |
| **5. Results** | 결과 제시 + held 속성 표시 | 유저 반응 관찰 (클릭, 체류) |
| **6. Feedback** | "결과가 어떠셨나요?" + 이메일 수집 | 만족도, 불만족 원인 태그, 초기 유저 확보 |

### 핵심: 각 스텝이 검색 공간을 수렴시킨다

```
전체 상품 DB (26,000+)
  → Step 1: 카테고리 필터 → ~3,000
  → Step 2: 속성 확정 (subcategory + color + fit) → ~500
  → Step 3: Hold hard filter → ~50-100
  → Step 4: 유사도 + 가격 범위 → 10-20개 최종 결과
```

Daydream은 같은 양의 정보를 **한 번의 자유 프롬프트**에서 추출하려 한다. 우리는 **6번의 쉬운 질문**으로 나눠서 얻는다. 유저의 인지 부담은 낮아지고, 시스템이 받는 신호의 정확도는 올라간다.

## 4. Daydream과의 구조적 차이

| 축 | Daydream | portal.ai |
|---|---------|-----------|
| **전제** | 유저가 취향을 언어로 표현할 수 있다 | 유저가 취향을 언어로 표현할 수 없다 |
| **입력** | "원하는 걸 말해봐" (자유 프롬프트) | "이 옷에서 뭐가 중요해?" (레퍼런스 기반 질문) |
| **해석** | 다중 모델 앙상블이 유저 의도 추측 | 유저가 직접 속성 확인·선택 → 추측 불필요 |
| **검색 방향** | 대화할수록 선택지 확장 (발산) | 질문할수록 조건 축소 (수렴) |
| **실패 모드** | 디자인 희석 — "비슷한데 다른 게 계속 나옴" | 범주 유지 — hold된 속성은 절대 이탈 안 함 |
| **UX 모델** | 채팅 (자유도 높음 → 뭘 써야 할지 모름) | 단계별 선택 (자유도 낮음 → 답하기 쉬움) |

### 왜 Daydream이 이 방식을 택하기 어려운가

기술적으로 Daydream도 Q&A 방식을 만들 수 있다. 그러나 **제품 구조와 비즈니스 모델이 충돌**하기 때문에 선택하지 않을 가능성이 높다.

**비즈니스 모델 충돌** — Daydream은 200+ 리테일 파트너의 상품을 넓게 노출하고 판매 수수료(~20%)를 받는 구조다. 결과를 10~20개로 좁히는 Q&A는 노출 수를 줄여서 파트너에 대한 가치제안과 정면으로 충돌한다. 넓게 보여줘야 돈을 버는 구조에서 좁히는 UX를 택할 인센티브가 없다.

**데이터 해자의 방향이 다르다** — Daydream이 쌓는 데이터는 자유 대화 로그다. 우리가 쌓는 데이터는 "어떤 속성을 hold하고, 어떤 걸 release했는지"의 속성 수준 취향 데이터다. 후자는 유저의 취향을 구조화된 형태로 축적하기 때문에, 개인화 단계에서 훨씬 직접적으로 활용할 수 있다. 서로 다른 종류의 데이터를 쌓고 있으므로, 시간이 지날수록 양쪽의 강점 영역이 분기한다.

> **근거**: 24명 인터뷰 전수 분석, 송진우 솔루션 인터뷰, GABI UX 분석, Daydream 비즈니스 모델 분석
> **상세**: [product-direction-qa-synthesis Part 4-7](26-04-13-product-direction-qa-synthesis.md), [daydream-competitive-analysis](26-04-12-daydream-competitive-analysis.md)

## 5. 기술 아키텍처

### 설계 원칙

세 가지 핵심 결정과 그 근거:

**1. Hold = Hard Filter (점수가 아닌 통과/탈락)**

유저가 "블랙 컬러"를 hold하면, 블랙이 아닌 상품은 점수와 무관하게 **무조건 제외**된다. "거의 블랙" 같은 유사 매칭을 하지 않는다.

> 근거 — 송진우: "재검색 시 내가 처음에 검색한 범주를 안 벗어나는 게 중요"
> 구현: `passesLockedFilter()` — held 속성 하나라도 불일치하면 탈락

**2. 구조화 매칭 우선, 벡터는 보조**

상품 검색의 1차 로직은 enum 기반 속성 비교(subcategory, color, fit, fabric, season, pattern)다. 벡터 임베딩은 이를 보완하는 2차 시그널로 설계한다.

> 근거 — ASEN 연구(IEEE/AAAI): 글로벌 임베딩은 fine-grained 속성(앞코 쉐입, 스티칭 디테일)을 놓침. FashionCLIP SOTA의 Text-to-Image Recall@1 = 12%에 불과. 인터뷰 24명 중 구매 결정 1순위가 "핏/실루엣/디테일"(20명) — 정확히 임베딩이 약한 영역.
> 구현: 10개 차원 가중합 스코어링 (subcategory 0.25 + colorFamily 0.20 + style gradient 0.30 + fit/fabric/season/pattern 각 0.15 + brandDna 0.20 + moodTags 0.15)

**3. 답할 수 있는 질문만 (슬라이더 X, 자유입력 X)**

유사도 조절을 0.0~1.0 슬라이더가 아닌 3단계 선택지(거의 똑같이 / 비슷하면 OK / 다른 스타일도)로 제공한다.

> 근거 — GABI 패턴: "몇 명이서 사세요?"처럼 간접적이고 답하기 쉬운 질문이 정확한 스펙 입력보다 완주율이 높음. 송진우: "카테고리를 아예 더 명확하게 받을 수 있도록 하는 게 좋은듯. 자유도가 높다보니 어떻게 검색해야 할지 고민."
> 구현: `SimilarityLevel` type (`tight` → tolerance 0.0, `medium` → 0.5, `loose` → 1.0)으로 매핑

### 현재 아키텍처 (Layer 1: Structured Matching)

```
┌─ 유저 입력 ─────────────────────────────────────────────┐
│  이미지 / 텍스트 프롬프트                                 │
└────────────────────────┬────────────────────────────────┘
                         ▼
┌─ AI 분석 (1회) ────────────────────────────────────────┐
│  Vision LLM → 아이템 분해 + 속성 추출                   │
│  (category, subcategory, fit, fabric, colorFamily,     │
│   season, pattern, styleNode, moodTags)                │
└────────────────────────┬───────────────────────────────┘
                         ▼
┌─ 6-Step Q&A (유저 상호작용) ──────────────────────────┐
│  Confirm → Hold → Conditions                          │
│  유저가 속성 확인·수정·선택 → 검색 조건 확정             │
└────────────────────────┬───────────────────────────────┘
                         ▼
┌─ 검색 엔진 ────────────────────────────────────────────┐
│  3-Path 검색 (AI분석 매칭 + 이름 키워드 + 직접 검색)     │
│  → passesLockedFilter (hard filter)                    │
│  → 10차원 가중합 스코어링                                │
│  → 다양성 캡 (브랜드 max 2, 플랫폼 max 3)              │
│  → tight=10 / medium=15 / loose=20개 결과              │
└────────────────────────┬───────────────────────────────┘
                         ▼
┌─ 피드백 루프 ─────────────────────────────────────────┐
│  결과 제시 → 유저 평가 (rating + tags + comment)       │
│  → 데이터 축적 → 추천 품질 개선 사이클                   │
└───────────────────────────────────────────────────────┘
```

**스택**: Next.js 16 (App Router, Vercel) + Supabase PostgreSQL + Cloudflare R2

**데이터 규모**: 26,000+ 상품, 22개 편집샵/브랜드몰 크롤링

### 진화 경로 (Layer 2 → Layer 3)

#### Layer 2: Hybrid Search (단기)

구조화 매칭의 한계 — 한국어 패션 슬랭("무지 맨투맨", "기모 후리스")을 수동 매핑(korean-vocab.ts, 115+항목)에 의존하고 있다. 신조어가 나올 때마다 수동 추가가 필요하며 확장에 한계가 있다.

**해결: 벡터 임베딩을 2차 시그널로 추가**

```
검색 쿼리
  → 구조화 매칭 (기존: enum exact match)        ─┐
  → 텍스트 임베딩 유사도 (BGE-M3, 한국어 네이티브) ─┼→ 하이브리드 스코어
  → 이미지 임베딩 유사도 (FashionSigLIP)          ─┘
  → Reranking (도메인 특화 리랭커)
  → 최종 결과
```

- **텍스트 임베딩**: BGE-M3 (multilingual, 한국어 네이티브) — "무지"와 "solid"가 수동 매핑 없이도 임베딩 공간에서 가까이 위치
- **이미지 임베딩**: FashionSigLIP (1M+ 패션 상품 학습, FashionCLIP 대비 +57% recall) — 이미지 ↔ 이미지 직접 유사도 비교
- **하이브리드 스코어링**: `structured_score × 0.6 + vector_score × 0.4` — 구조화 매칭의 정밀도를 유지하면서 어휘 한계 보완
- **벡터 DB**: pgvector (Supabase 내장) — 26K 규모에서 HNSW 인덱스로 <20ms 응답, 별도 인프라 불필요

#### Layer 3: Agentic RAG (중기)

현재 Q&A 플로우는 1회성이다 — 결과를 보고 "좀 더 캐주얼하게" 같은 상대적 지시를 처리할 수 없다. 이전 맥락을 기억하고 점진적으로 조정하는 대화형 에이전트가 필요하다.

**해결: LangGraph 기반 대화형 추천 에이전트**

```
┌─ Vercel (Next.js) ──────────────────────────────────┐
│  프론트엔드 + 기존 API (분석, 피드백)                  │
└───────────────────┬─────────────────────────────────┘
                    │ HTTP / SSE
┌───────────────────▼─────────────────────────────────┐
│  AI Server (FastAPI)                                │
│                                                      │
│  ┌─ LangGraph ────────────────────────────────────┐ │
│  │  의도 분석 → 검색 전략 → 하이브리드 검색         │ │
│  │  → 리랭킹 → 결과 생성                           │ │
│  └────────────────────────────────────────────────┘ │
│                                                      │
│  ┌─ 벡터 검색 ──────┐  ┌─ LLM 라우팅 ────────────┐ │
│  │  pgvector        │  │  LiteLLM               │ │
│  └──────────────────┘  └─────────────────────────┘ │
│                                                      │
│  ┌─ 관찰성 ─────────────────────────────────────┐   │
│  │  LangSmith — trace 시각화 + 프롬프트 A/B     │   │
│  └──────────────────────────────────────────────┘   │
└───────────────────┬─────────────────────────────────┘
                    │
        ┌───────────▼──────────────┐
        │  Supabase (PostgreSQL)   │
        │  + pgvector extension    │
        └──────────────────────────┘
```

**핵심 변화: Query Enhancement**

현재: "좀 더 캐주얼하게" → 맥락 없이 새로 키워드 추출 → 비슷한 결과 반복

개선: "좀 더 캐주얼하게" + 이전 검색이 "미니멀 울 코트"였음을 기억 → "미니멀 무드이지만 소재는 면/저지, 핏은 릴랙스드" → 의미 있게 다른 결과

**기술 스택 요약**:

| 영역 | 기술 | 이유 |
|------|------|------|
| AI 서버 | FastAPI | Python AI 생태계 접근 (LangGraph, 임베딩, 리랭킹) |
| 에이전트 그래프 | LangGraph | 대화 상태 관리 + 조건부 노드 분기 |
| 텍스트 임베딩 | BGE-M3 | 한국어 네이티브, 다국어 지원 |
| 이미지 임베딩 | FashionSigLIP | 패션 도메인 특화 (Apache 2.0) |
| 벡터 DB | pgvector → Qdrant (스케일 시) | 26K에 pgvector 충분, 100K+ 시 마이그레이션 |
| 리랭킹 | Cohere Rerank | 한국어 네이티브, 도메인 재정렬 |
| LLM 라우팅 | LiteLLM | 멀티 프로바이더 관리 + 비용 최적화 |
| 관찰성 | LangSmith | 추천 흐름 추적 + 프롬프트 A/B 테스트 |

## 6. 현재 구현 상태 (2026-04-14)

| 영역 | 상태 | 비고 |
|------|------|------|
| 6-Step Q&A 플로우 | ✅ 완료 | input → confirm → hold → conditions → results → feedback |
| 속성 확인 + 수정 | ✅ 완료 | 커스텀 드롭다운, enum 기반 선택지 |
| Hold hard filter | ✅ 완료 | 0~3개 속성 hold, passesLockedFilter |
| 3단계 유사도 | ✅ 완료 | tight=10 / medium=15 / loose=20개 결과 |
| 빈 결과 release 추천 | ✅ 완료 | pickUnlockSuggestion 연동 |
| EN/KO 실시간 전환 | ✅ 완료 | 헤더 토글, enum 한글 디스플레이 |
| 피드백 + 이메일 수집 | ✅ 완료 | rating + tags + comment + email → Supabase |
| 검색 엔진 v4 | ✅ 완료 | gradient scoring + 브랜드 DNA + 한국어 어휘 115+ |
| 검색 디버거 (어드민) | ✅ 완료 | score breakdown 시각화 |

## 7. 로드맵

| Phase | 내용 | 기술 추가 | 핵심 지표 |
|-------|------|----------|----------|
| **1 (현재)** | 6-Step Q&A + enum 매칭 + 피드백 수집 | — | 플로우 완주율, positive rate |
| **2** | 벡터 임베딩 도입 (텍스트 + 이미지), 하이브리드 스코어링 | pgvector + BGE-M3 + FashionSigLIP | 검색 recall, 한국어 슬랭 매칭률 |
| **3** | AI Server + Agentic RAG, 대화형 재탐색 | FastAPI + LangGraph + LiteLLM | 재탐색 정확도, 맥락 반영률 |
| **4** | 리랭킹 + 관찰성, 품질 자동화 | Cohere Rerank + LangSmith | 추천 품질 회귀 감지, 비용/쿼리 |
| **5** | 취향 프로필 축적, 세션 간 개인화 | hold 패턴 학습 + 유저 프로필 DB | 재방문율, 세션당 전환율 |

### Phase 간 핵심 전환점

- **1→2**: "한국어로 검색하면 안 나온다"는 피드백이 임계치를 넘을 때
- **2→3**: "결과 보고 조정하고 싶은데 방법이 없다"는 피드백이 임계치를 넘을 때
- **3→4**: 쿼리 볼륨이 일일 1,000건을 넘어 수동 품질 관리가 불가능할 때
- **4→5**: 재방문 유저 비율이 30%를 넘어 개인화의 ROI가 발생할 때

각 Phase는 이전 Phase의 유저 피드백 데이터가 전환 근거가 된다. Step 6(피드백 수집)이 **모든 Phase 전환의 신호 수집기** 역할을 한다.
