# Part 1. 서문 — 왜 이 논의가 필요했는가

## 1.1 배경의 긴장

portal.ai는 그동안 "이미지 한 장으로 크로스플랫폼 패션 검색"이라는 가치 제안으로 개발을 진행해왔다. 그러나 2026년 4월 시점에서 팀 내부에 다음과 같은 전략적 긴장이 발생했다:

1. **시장에 Daydream이라는 거대한 선례가 존재**한다. $50M 시드, 61명 팀, 10,000+ 브랜드 커버리지, TIME Best Inventions 2025 선정. "AI 기반 패션 쇼핑 에이전트"라는 카테고리 자체를 선점한 상태.
2. **팀이 구상한 "재탐색 자동화" 방향이 충분한 차별점이 아닐 수 있다**는 인식. 구조적 개편일 뿐이라 Daydream이 마음만 먹으면 따라잡을 수 있다.
3. **유저에게 실제로 설득력 있는 차별점이 무엇인가에 대한 확신 부족**. PT로 포장해도 트랙션이 없으면 의미 없다.
4. **2인 팀이라는 제약**. Daydream과 같은 기술 arms race에서는 이길 수 없다.

## 1.2 팀이 가져온 근본 질문

팀원 간 디스코드 대화(2026-04-10)에서 도출된 질문:

> "Daydream은 어떻게 추천하고 있는지에 대해 설명할 수 있어야 하고, 우리는 어떻게 다른 방식으로 추천할 수 있을 것인지에 대해 말할 수 있어야 함."

이때 **Snapdeck**(PPT 대체 AI 슬라이드 서비스)의 사례가 레퍼런스로 언급되었다. Snapdeck의 차별화 공식:

- 기존 AI 슬라이드: **템플릿(미리 만든 구조)에 텍스트 채우기** → AI 티 남, 디자인 한정적
- Snapdeck: **LLM이 코드를 직접 써서 슬라이드를 그림** → 100% 유저 의도 반영
- 핵심 교훈: "기존 방식이 왜 그렇게 될 수밖에 없었는지"를 이해하고, **그 제약을 우회하는 다른 메커니즘**을 찾음

**팀의 도전 과제**: 패션 검색에서 Snapdeck 같은 "다른 메커니즘"은 무엇인가?

## 1.3 이 문서의 목적

단순한 결론 공유가 아니라, **결론에 이르기까지의 모든 갈림길과 선택 근거를 기록**하는 것이 목적이다. 이유는:

- 기획자가 이 방향을 **내재화**하려면 "왜 다른 길은 안 되었는지"를 알아야 한다
- 향후 아키텍처 설계 시 **어떤 제약이 본질적이고 어떤 것이 우연적**인지 구분해야 한다
- 상황이 바뀌었을 때 **어느 지점에서 재고할 수 있는지** 명시해둔다

# Part 2. 출발점 — 팀이 가져온 초기 가설

## 2.1 포지셔닝 가설

팀 논의(2026-04-10)에서 제안된 초기 포지셔닝:

> "Daydream은 **넓고 세련된 패션 쇼핑 에이전트**다.
> 우리는 **아주 구체적인 취향을 끝까지 이해하는 니치 취향 에이전트**다."

이 포지셔닝의 근거:
- Daydream은 모든 유저를 위한 매스 서비스를 지향 → 깊이 부족
- 서브컬처/디자이너 브랜드에 대한 감도는 한국 니치 유저가 더 높을 수 있음
- 매스가 아닌 니치에 집중하면 "취향의 해상도"에서 압도 가능

## 2.2 골든 유스케이스

팀이 정의한 "우리가 해결하고 싶은 가장 날카로운 문제":

> "발렌시아가 마이애미 더비 같은 제품인데, 그것보다 저렴한 제품을 찾아줘"

팀이 이 쿼리를 확장한 버전:
> "발렌시아가 마이애미 더비(앞코가 긴 쉐입이 섹시한 더비슈즈이며, 앞코의 쉐입은 아주 약간의 스퀘어토가 있고 너무 스퀘어토가 각지지 않아서 클래식하면서도 약간은 캐주얼해 청바지와도 매치할 수 있는 것) 같은 제품인데, 그것보다 저렴한 제품을 찾아줘"

## 2.3 초기 가설의 핵심 메커니즘

팀이 도출한 차별화 메커니즘:

> Daydream식 구조: 유저 프로필 + 자연어 질의 + 대형 카탈로그 매칭 + refinement
>
> 우리가 가야 할 구조: **레퍼런스 제품 분석 + 취향 속성 추출 + 속성 우선순위화 + 서브컬처 브랜드 그래프 탐색 + 대체 가능성 설명**

질문의 중심이
> "뭘 찾으세요?"가 아니라
> **"당신이 이 제품에서 진짜 좋아한 건 정확히 무엇인가?"**
로 바뀌어야 한다.

## 2.4 이 시점의 미해결 질문

이 가설은 방향성이지만 구체적 메커니즘이 없었다. 해결해야 할 질문:

1. "취향 속성 추출"을 어떻게 할 것인가?
2. "속성 우선순위화"를 어떻게 구현할 것인가?
3. "브랜드 그래프"를 어떻게 구축할 것인가?
4. 이 모든 것이 Daydream이 하는 것과 어떻게 다른가?

**→ 이 질문들을 하나씩 검증하는 것이 이후 논의의 여정이었다.**

# Part 3. 경쟁사 심층 이해 — Daydream의 구조적 본질

## 3.1 Daydream을 이해하지 않고 차별화할 수 없다

"Daydream과 다르게 한다"는 말이 의미가 있으려면, Daydream이 정확히 어떻게 작동하는지 먼저 알아야 한다. 그래서 Daydream의 인터뷰(The Interline Podcast, 2025.12), 언론 보도, 채용 공고, LinkedIn을 종합 분석했다.

## 3.2 Daydream의 기술 아키텍처

### Ensemble of Experts
Maria Belousova(CTO) 직접 언급:
> "Our current state of the system is actually an ensemble of experts"

Fortune 기사(2025.06.25):
> "Daydream's results are influenced by **around a dozen small language models**, each of which addresses certain attributes of merchandise, like the color or fabric."

구조:
```
유저 쿼리/이미지 입력
  ↓
12개 전문 모델 (formality, season, color, occasion, bodytype, silhouette ...)
  ↓ (각 모델이 의견 제시 → 합의 집계)
Comprehensive Intent Understanding
  ↓
Fashion Knowledge Graph를 통한 "shopper vocabulary → merchant vocabulary" 번역
  ↓
Hybrid Search (Lexical + Vector + Image)
  ↓
결과
```

### 검색 방식 4가지
1. **Lexical Search**: 전통적 키워드 매칭 (Elasticsearch 추정)
2. **Hybrid Search**: Lexical + Vector 결합
3. **Vector Search**: PGVector / Qdrant / Weaviate 중 하나 (채용 공고 명시)
4. **Image Search**: Visual Intelligence 기반 (CLIP/FashionCLIP 계열 추정)

### 데이터 파이프라인
- 200+ 리테일 파트너로부터 피드 수신 + 크롤링
- **Enrichment**: 피드에 없는 잠재 속성을 AI로 추출 (portal.ai의 `product_ai_analysis`와 동일 개념)
- **Fashion Knowledge Graph**: 소비자 언어 ↔ 상품 속성 양방향 매핑
- **사내 패션 전문가 팀**이 훈련 데이터 직접 생성

## 3.3 Daydream 방식의 구조적 본질

**핵심 발견**: Daydream이 하는 것은 **"쇼퍼 언어 → 머천트 언어" 번역 사전(dictionary)을 사람이 만드는 일**이다.

예시:
```
"revenge dress" → {category: dress, mood: bold, occasion: wedding,
                   style: dramatic, vibe: statement}
"Saltburn style" → {era: 1990s-2000s, aesthetic: british-preppy,
                    palette: muted-pastels}
```

이 사전을 유지하기 위해:
- 패션 전문가 팀이 계속 항목 추가
- 훈련 데이터를 수동으로 생성
- 각 쇼퍼 표현에 대응하는 머천트 속성을 수작업 매핑

## 3.4 이 구조의 전제 조건

Daydream 방식이 작동하려면 다음이 필요하다:

| 조건 | Daydream이 가진 것 |
|------|-------------------|
| 자본 | $50M 시드 |
| 인력 | 61명, 전문가 포함 |
| 데이터 | 200만 상품, 200+ 리테일 파트너 피드 |
| 브랜드 인지도 | Julie Bornstein(Stitch Fix 전 COO) 네트워크 |
| 운영 시간 | 2023년 설립, 2025년 6월 런칭까지 2년 |

## 3.5 portal.ai가 같은 판에서 경쟁할 수 없는 이유

**모든 리소스 차원에서 열세**.

| 조건 | portal.ai | Daydream | 격차 |
|------|-----------|----------|------|
| 자본 | 부트스트랩 | $50M | 무한대 |
| 인력 | 2명 | 61명 | 30배 |
| 데이터 | 45K 상품 | 200만 상품 | 44배 |
| 운영 기간 | 짧음 | 2년+ | — |

**결론**: Daydream이 만든 사전(KG)과 같은 구조로 경쟁하면 체급 차이로 반드시 진다. **다른 메커니즘이 필요하다.**

하지만 "다른 메커니즘이 뭐냐?"는 질문은 이 시점에서 답할 수 없었다. **유저가 실제로 무엇을 하고 싶어하는지부터 확인해야 했다.**

# Part 4. 유저 실체 확인 — 24명 인터뷰가 말해준 것

## 4.1 왜 인터뷰 분석이 결정적이었는가

Daydream의 구조를 이해한 상태에서 답해야 할 질문:
> "Daydream이 풀고 있는 문제와, 유저가 실제로 겪는 문제가 같은가?"

만약 같다면, 우리는 같은 문제를 다른 방식으로 푸는 것 → 열화판 가능성.
만약 다르다면, Daydream이 놓치고 있는 공백이 존재 → 차별화 가능.

팀이 진행한 55명 유저 인터뷰 중 Notion에 정리된 24명의 내용을 **전수 분석**했다.

## 4.2 발견의 시작점 — 24명 전원 공통 패턴

### 텍스트 검색으로 시작하는 사람: **0명**

모든 유저가 **비주얼 브라우징**(인스타 릴스, 유튜브 하울, 핀터레스트)에서 시작한다.

예시 인용:
> **허자연**: "유튜브에 봄옷, 인스타에 봄옷이라고 침. 예쁜 언니들이 봄 옷 하울을 찍는다."
> **윤병현**: "릴스 쭉 봐. 내가 좋아하는 브랜드 신상에서 보고, 해당 사이트에도 들어가보고."
> **김지혜**: "인스타에서 코디, 핀터레스트 등에서 찾음."
> **송진우**: "인스타에서 워낙 많이 올려버리니까 탐색 측면에서는 효율적이라고 느꼈음."

### 시사점

Daydream의 주요 진입점 중 하나인 **"Chat-to-Shop"(자연어 대화)**는 한국 니치 유저의 행동 패턴과 어긋난다. 그들은 **이미지를 먼저 본다.**

portal.ai의 기존 강점인 "이미지 업로드 → 분석"은 방향이 맞다. 다만 이것만으로는 충분하지 않다 (뒤에 이유 설명).

## 4.3 재탐색 행동 — 3가지 패턴으로 수렴

재탐색 = "마음에 드는 아이템을 발견한 후 대안/비교/확신을 찾는 과정".

팀이 해결하겠다고 했던 핵심 문제가 바로 여기다. 인터뷰를 보면 3가지 패턴으로 수렴했다:

### 패턴 A. 사람 추종형 (약 10명)
> **송진우**: "원하는 스타일을 입는 모델들을 셀렉을 많이 한다. 거기서 옷을 고른다. (무드가 맞는 사람에 맞춰 그 사람이 입은 옷을 구매)" — 효과 7/10
>
> **허자연**: "나와 체형이 비슷할 것 같고, 모델보다 잘 어울릴 것 같으면 본다"
>
> **유다연**: "잘 보고 있는 인플루언서를 타고 타고 들어가는 게 성공을 잘 하는 듯"

**핵심 행동**: 취향을 언어화할 수 없으니, **"무드가 맞는 사람"을 프록시로 사용**.

### 패턴 B. 플랫폼 순회형 (약 8명)
> **오민지**: "같은 상품이 모든 플랫폼에 있으니, 모든 리뷰를 잘 찾아보고 싶어서 각 쇼핑플랫폼마다 확인하는 편"
>
> **최윤영**: "품번 검색해서 여러 플랫폼 후기 싹다 크로스체크"
>
> **진혜원**: "무신사 랭킹 + 좋아하는 브랜드 신상 탐색"

**핵심 행동**: 여러 플랫폼을 다니며 가격/리뷰/핏 정보를 교차 확인.

### 패턴 C. 오프라인 확인형 (약 6명)
> **윤원도**: "직접 보고 느낌이 와야 살 수 있어서 오프라인으로 자주 구매한다."
>
> **김해나**: 빈티지는 "이거 완전 내껀데" 하면 즉결 (입어보고 핏 맞으면 바로 결제)
>
> **김규리**: "온라인에서 옷을 사기 어려운 이유가 핏/사이즈에서 비롯함."

**핵심 행동**: 온라인은 발견만, 실제 구매는 매장.

### 공통 관찰
모든 패턴에서 공통적으로:
- 탐색 기간 **1~4주**
- 하루 **30분~2시간** 투자
- "**숙제처럼 느껴진다**" (오민지)
- "**귀찮다. 안사고 말지**" (최윤영)

## 4.4 판단 기준 — 24명 우선순위 집계

각 인터뷰에서 "구매 결정 시 무엇을 보는가"를 물었고, 응답 빈도를 집계했다:

| 순위 | 기준 | 언급 빈도 | 비고 |
|------|------|---------|------|
| **1위** | **핏 / 실루엣 / 사이즈** | **~20명** | 압도적 1위 |
| 2위 | 디자인 / 디테일 | ~18명 | 기장, 넥라인, 패턴 등 |
| 3위 | 소재 / 질감 | ~12명 | 보풀, 세탁, 촉감 |
| 4위 | 기존 옷과의 매칭 | ~10명 | "내 옷장과 어울리는가" |
| **5위** | **가격** | **~8명** | **의외의 후순위** |
| 6위 | 리뷰 / 후기 | ~12명 | "나와 비슷한 체형" |

### 이것이 왜 중요한가

팀의 초기 가설("발렌시아가 마이애미 더비 같은데 **더 저렴한** 것")은 가격을 핵심 value prop에 두었다. 그러나 인터뷰는 다른 이야기를 한다:

> 고감도 유저일수록 가격은 후순위이며, **"핏과 디테일이 정확히 맞는가"**가 절대적 우선 기준이다.

- 송진우: "가격, 사이즈, 색상은 크게 고려하지 않는다. 아이템만 괜찮으면 산다."
- 윤원도: "디자인이 독특하면 가격 무관"
- 하유빈: "10만원 정도는 그냥 쓰고, 20만원 중반대까지도 개이쁘면 바로 산다"

**→ 팀 가설의 "더 저렴한"은 value prop의 핵심이 아닐 수 있다. 진짜 핵심은 "내가 원하는 것을 정확히 찾아주는 것".**

## 4.5 가장 중요한 발견 — "취향 언어화 실패"

모든 인터뷰에서 공통적으로 발견된 Pain Point:

> **유저는 자신의 취향을 언어로 표현할 수 없다.**

### 직접 인용
> **허자연**: "추구미는 있는데 패션 세계 안에서 엄청난 이름이 있을거 아님. 페미닌같은거. **용어도 잘 모르고 진짜 딱 맞는 용어가 있을텐데 그걸 모른다. 프롬프트 쓰기도 어려울 수 있다.**"
>
> **이장혁**: "구체적인 게 떠오르지 않은 상태에서 검색을 함. 번화가에서 입을 수 있는 그런 패션 이렇게 침. 단정한 것만 나올 때도 있고 유니크한 것만 나올 때도 있고 해서 나의 취향과 안 맞는다는 생각이 든다. **중간점이 없다는 느낌이 듬.**"
>
> **김지혜**: "플랫폼을 보면 아우터, 상의, 바지 이렇게 분류를 해놓잖아요. 근데 그 안에서 내가 원하는 추구미가 다르잖아요. **내가 원하는 추구미대로 대분류를 먼저 해줬으면 좋겠음.**"
>
> **여지혜** (무신사 업계 종사자): "자기가 지금 찾고 있는 **네이밍 자체를 모르는 사람들이 많음. 뭔가를 가지고 싶은데 그게 뭔지를 정확하게 모름.**"

### 이것이 의미하는 것

Daydream은 "revenge dress for a wedding in Paris"처럼 유저가 자연어로 표현한 복잡한 쿼리를 해석하는 데 $50M을 쓰고 있다. 그런데:

> **한국 니치 유저는 그런 프롬프트를 쓸 수 없다.**

그들은 "추구미는 있는데 용어를 모른다." 프롬프트 입력 자체가 불가능에 가깝다.

**→ Daydream은 "유저가 자연어로 쿼리를 쓸 수 있다"는 전제를 가지고 있지만, 그 전제 자체가 한국 니치 유저에게는 맞지 않을 수 있다.**

이 발견은 나중에 최종 결론의 결정적 단서가 된다.

## 4.6 송진우 솔루션 인터뷰 — 결정타

송진우는 특별히 중요했다. 이유:
1. Daydream을 **실제로 사용**한 유저
2. portal.ai 프로토타입도 **직접 테스트**한 유저
3. 둘을 비교하며 피드백을 명확히 남김

### Daydream에 대한 피드백
> "데이드림에서 검색 시 재탐색 시 **가격은 낮아지나, 원하는 디자인이 자꾸 희석되는 느낌**"
>
> "재검색 시 **내가 처음에 검색한 범주를 안 벗어나는 게 중요**할 것 같다고 느낌"
>
> "디자인은 이쁜데 비싸고, 재탐색이 어렵다."
>
> "**너무 다양하게 추천해서** '괜찮은데 싶다가도 이게 필요한 게 아니라서 망설여짐'"
>
> "나는 이미 데님으로 정했고 가격대로 요구를 했고 컬러도 보여줬는데, **오히려 선택지가 많아져서 불편하고, 차라리 좀 좁혀지는 편이 더 좋은 듯. 다른 게 뜨면 너무 피로하다.**"

### portal.ai 프로토타입에 대한 피드백
> "초반에 **프롬프트 외에 카테고리를 아예 더 명확하게 받을 수 있도록 하는게 좋은듯**. 너무 자유도가 높다보니 어떻게 검색해야 할지 고민이 된다."
>
> "**오히려 자유도를 낮췄으면 좋겠다**"

### 송진우가 직접 정의한 문제
> **"사용자는 '더 저렴한 제품'이 아니라, '내가 원했던 그 제품과 같은 결의 대안'을 원한다."**

### 시사점

이 인터뷰는 결론의 핵심 방향성을 정확히 제시했다:

1. **"자유도를 낮춰라"** — 자유 입력 방식의 근본적 문제점을 유저가 직접 지적
2. **"범주가 희석되지 않아야 한다"** — 재탐색 시 원래 결을 유지해야 한다
3. **"선택지가 많아지면 피로"** — 추천 결과를 좁혀야 한다
4. **"저렴한 것이 아니라 같은 결"** — 가격이 아닌 결(vibe) 매칭이 핵심

이 피드백들은 이후 모든 의사결정의 기준점이 된다.

# Part 5. 기술적 차별화의 검토와 기각

인터뷰에서 "자유도를 낮추고, 범주를 좁히고, 결을 유지하라"는 시그널을 받았지만, 이를 **기술적으로 어떻게 구현할지는 별개 문제**였다. 팀이 고민한 3가지 기술 옵션을 순서대로 검토했다.

## 5.1 옵션 A. LLM 기반 속성 매칭 고도화 + "정체성 점수"

### 제안 내용

레퍼런스 제품에서 AI Vision으로 속성을 추출하고, 우리 DB의 통계 분포를 이용해 **"정체성 점수(identity score)"**를 매긴다. 이 점수로 속성별 가중치를 정해 검색.

```
발렌시아가 마이애미 더비를 예로 들면:

속성                    값                    정체성 점수
───────────────────────────────────────────────────
앞코 쉐입              elongated square       0.95  ← 이게 정체성
더비 실루엣            low-cut, lace-up       0.80
캐주얼/클래식 밸런스    청바지 OK             0.60
가죽 소재              smooth leather         0.30
검정 색상              black                  0.10
```

정체성 점수 계산 원리:
> "이 카테고리에서 흔하지 않은 속성일수록, 이 제품의 정체성을 정의하는 속성이다."
> DB에 있는 더비슈즈 500개 중 square toe가 20%면 → 이 속성은 distinctive → 정체성 점수 ↑

### 검증: 이게 Daydream과 다른가?

**답: 다르지 않다.**

- Daydream도 LLM으로 쿼리에서 속성/키워드를 추출한다
- 전문가 팀은 이 추출 결과의 매핑 정확도를 높이기 위해 존재한다
- 우리가 "정체성 점수"로 가중치를 매긴다고 해도, 기본 메커니즘(속성 추출 → 매칭)은 동일

### 기각 이유

1. **열화판 문제**: Daydream이 하는 것과 구조적으로 같음. 전문가 없이, DB도 작고, 매핑도 부족한 상태로 이기는 건 불가능.
2. **DB 요구 사항**: "정체성 점수"가 의미 있으려면 카테고리별 수백 개 상품이 필요. 45K 상품 중 category 필드가 제대로 채워지지 않은 상태.
3. **enum 세분화 비용**: toe_shape, collar_type 같은 fine-grained 속성을 정의하고 모든 상품에 태깅하는 건 Daydream의 전문가 팀이 하는 일과 동일.

### 팀원의 결정타 지적
> "데이드림도 프롬프트에서 dna 같은 키워드를 안 뽑지 않을 거 아냐? 뽑았는데 거기서 더 정확도를 높이기 위해, 즉 내부적으로 매핑을 확실하게 하기 위해서 팀이 패션 도메인 워딩 매핑을 하는 것일 꺼라는 말이지?? 근데 우리가 그거를 이 구조로 이길 수가 있음?"

**→ 맞는 지적. 같은 판에서 이길 수 없음. 옵션 A 기각.**

## 5.2 옵션 B. 이미지 임베딩 직접 비교 (FashionCLIP)

### 제안 내용

속성 추출 단계를 건너뛰고 이미지 임베딩을 직접 비교. 유저가 레퍼런스 이미지를 주면 FashionCLIP으로 벡터화해서 우리 DB의 상품 이미지 벡터와 코사인 유사도 계산.

**핵심 주장**: "square toe"를 단어로 표현하지 못해도, 이미지 속에 그 정보가 담겨 있으므로 임베딩이 그걸 잡는다.

### 검증 1: Daydream도 이걸 하고 있는가?

**답: 그렇다.**

Daydream 채용 공고와 The Interline 팟캐스트에서 확인:
- **Vector Search** (PGVector / Qdrant / Weaviate 사용)
- **Image Search** (Visual Intelligence 기반, CLIP/FashionCLIP 계열 추정)
- **Embeddings** (다양한 데이터셋에 대한 embedding strategy)

즉, Daydream의 4가지 검색 방식 중 하나가 이미 이미지 임베딩이다.

### 검증 2: FashionCLIP의 실제 성능은?

공개된 SOTA 모델(Marqo-FashionSigLIP, 2026.02 최신)의 벤치마크:

| 지표 | 값 | 해석 |
|------|-----|------|
| Text-to-Image Recall@10 | **34%** | 10개 결과 중 관련 있는 건 3.4개 |
| Text-to-Image Recall@1 | **12.1%** | 첫 결과가 정답인 경우는 12% |
| Category-to-Product Precision@1 | 76% | 카테고리 수준 매칭은 강함 |

**→ 자유형 텍스트 검색은 생각보다 약함**.

### 검증 3: Fine-grained 디테일을 잡는가?

| 세부 속성 | 캡처 수준 |
|-----------|----------|
| 카테고리 (jacket, sneaker) | 우수 |
| 색상 | 우수 |
| 전반적 스타일 | 양호 |
| 실루엣 (oversized, slim) | 보통 |
| **패브릭 텍스처** | **미흡** |
| **디테일 (toe shape, 단추, 스티칭)** | **미흡** |

근본 원인: ViT 입력이 224x224px → 미세한 디테일 캡처 불가. 우리가 가장 잡고 싶은 "앞코 쉐입" 같은 것은 **임베딩이 놓친다**.

### 기각 이유

1. **Daydream도 사용 중** → 차별화 요소 아님
2. **Fine-grained 성능 한계** → 우리가 타겟하는 "니치 취향"의 핵심(미세 디테일)을 못 잡음
3. **한국어 미지원** → 영어 전환 필요
4. **DB 커버리지 문제** → 유사 상품이 없으면 임베딩이 우수해도 결과 0건

**→ 이미지 임베딩은 보조 도구로 쓸 수는 있지만, 이걸로 Daydream과 차별화할 수는 없다. 옵션 B 기각.**

## 5.3 옵션 C. 2개 카테고리에 깊이 집중

### 제안 내용

신발 + 하의 2개 카테고리에만 집중. 다른 카테고리는 포기하되, 이 카테고리에서는 세계 최고 수준의 속성 세분화와 매칭 정확도 달성.

논리: Daydream은 200만 상품을 얕게 커버, 우리는 2개 카테고리를 극도로 깊게 커버.

### 검증: 이게 지속 가능한 moat인가?

**답: 아니다.**

- 기술 격차는 복제 가능. Daydream이 마음먹고 한국 시장에 들어오면, 우리가 2개 카테고리에 쌓은 깊이를 2배 인력으로 6개월 만에 따라올 수 있음.
- 팀원의 지적: "어쨌든 우리는 지금 DB랑 인력 자체가 부족한 상황인데, 그렇게 딥하게 기술력으로 판다고 해도 데이드림에서 이걸 한다면 따라잡히지 않을까?"

### 추가 리스크

1. **DB 커버리지**: 45K 중 신발/하의가 몇 개인지도 현재 구조로는 파악 불가
2. **enum 태깅 비용**: toe_shape, sole_type 등 fine-grained enum을 정의하고 모든 상품에 태깅하려면 상당한 수작업
3. **범용성 손실**: 2개 카테고리만 커버하면 "패션 에이전트"가 아닌 "신발 검색기"로 축소될 위험

**→ 옵션 C 기각.**

## 5.4 기술적 차별화의 구조적 한계

3개 옵션을 모두 기각한 뒤 도달한 결론:

> **"모든 기술 경로가 Daydream의 판 안에 있다."**

기술적 우위는:
- 더 많은 자본으로 따라잡을 수 있고
- 더 많은 인력으로 따라잡을 수 있고
- 더 많은 시간으로 따라잡을 수 있다

Daydream은 이 3가지를 모두 가졌다. 우리는 모두 부족하다. **기술 경쟁에서는 질 수밖에 없다.**

필요한 것은 **"판 자체를 바꾸는 것"** — Snapdeck이 템플릿 방식을 버리고 코드 생성으로 전환한 것처럼.

하지만 이 시점에서 "판을 바꾸는 게 구체적으로 뭔가"는 아직 불명확했다. **다른 방향을 탐색해야 했다.**

# Part 6. 풀 저니 방향의 검토와 유보

> "유저가 이미지를 업로드하면 그걸 기반으로 뽑아주고, 추천된 아이템에서 또 다른 아이템을 추천해주거나 핏감 사이즈 등을 개인화해서 알려주거나.. 조금 더 유저 친화적으로 가보는 게 낫지 않을까 싶기도 한데... 탐색부터 구매, 내 옷장 관리까지 모두 진행해주는 거지."

**핵심 발견**: 모든 열이 O인 서비스는 **단 하나도 없다**. 빈 공간(white space) 존재.

### 구조적 장벽

왜 아무도 풀 저니를 못 만드는가:

1. **비즈니스 모델 충돌**
   - 발견 앱: affiliate 커미션 → 구매 후 경험 구축 인센티브 없음
   - 옷장 앱: "안 사는 게 좋다" 철학 → 커머스와 본질 충돌
   - 커머스 플랫폼: 자사 재고 판매 목적 → 크로스플랫폼과 대립

2. **데이터 사일로**
   - True Fit: 핏 데이터 최강, 스타일 없음
   - Daydream: 스타일 최강, 구매 완료 데이터 없음
   - 옷장 앱: 소유 데이터만, 카탈로그 없음

3. **기술적 복잡도**
   - "250가지 빨간색" (Alta Menlo VC 인용) - 색상 하나도 복잡
   - 이미지 검색 + 자연어 + 옷장 + 핏을 동시에 = 10+ 특화 모델 필요

4. **콜드 스타트 문제**
   - 옷장 디지타이징이 극도로 높은 온보딩 비용

**→ 이 장벽들이 우리에게는 moat가 될 수 있다.**

### 검토 포인트 2: Alta의 약점
1. **Affiliate 모순**: 옷장 활용이 잘될수록 구매 감소 (실제 리뷰에서 포착됨: "shop recommendations reduced as they've added more to their closet")
2. **Recommendation Quality 미완성**: 7일 챌린지에서 "completely unsalvageable" 날 존재, 업무/더운 날씨 추천 약함

4. **기능 폭주**: closet + outfit + shopping + trip + avatar + social 전부 → 2인 팀이 따라하면 100% 실패

## 6.5 이 방향의 한계 — 왜 완전한 답이 아닌가

그러나 풀 저니 방향은 단기 차별화의 답이 되기 어려웠다:

1. **Alta가 선점 중**: 차별점이 "한국 니치 브랜드"로만 축소됨. 제품 철학 자체의 차별은 약함.
2. **옷장 디지타이징 온보딩**: Acloset 기준으로도 드롭률이 심각. 실행 리스크 큼.
3. **기능 폭주 위험**: 2인 팀이 closet + outfit + shopping + trip + avatar를 다 하면 100% 실패.
4. **재탐색/취향 해체 문제 미해결**: 옷장을 관리한다고 해서 송진우의 "디자인 희석" 문제가 풀리지 않음.
5. **Affiliate 모델 모순**: Alta가 겪는 문제를 우리도 그대로 겪음.

**→ 풀 저니는 "장기 지향점"으로는 유효하지만, "단기 차별화의 핵심"이 될 수 없다.**

## 6.6 이 시점의 상황

3가지 기술 옵션 기각 + 풀 저니 방향도 유보.

"그러면 진짜 답은 뭔가?"라는 질문이 남았다.

이 지점에서 팀원이 **올웨이즈의 GABI(가전비서)**를 힌트로 던졌다.

# Part 7. 패러다임 전환의 발견 — GABI + 송진우 피드백의 합류

## 7.1 GABI를 단서로 본 이유

팀원의 제안:
> "올웨이즈 가전비서처럼 앞단에서 받는 것을 체계화해서 ㅈㄴ게 파고들어서 진짜 유저가 원할 만한 결과물을 내보내는 건 어떨까? 지금은 너무 데이드림이랑 유사한 것 같아서 말이야."

GABI가 가진 것은 "자유로운 대화"가 아니라 **"체계적인 단계별 질문"**이다.

## 7.2 GABI의 실제 플로우

```
1. 온보딩 (/onboarding)
2. 견적서 업로드 (/consultation/upload)
   → 기존 견적서 이미지를 올리면 AI가 자동 파싱
3. OCR/AI 견적 파싱 (/consultation/parsing)
   → 품목, 가격, 모델명 자동 인식 (16+ 카테고리)
4. 라이프스타일 Q&A (/quote-agent/lifestyle-questions)
   → 가족 수, 주거 크기, 요리 빈도, 브랜드 선호, 우선순위 설문
5. 구매 목적 상담 (/consultation/purpose)
   → 구매 계기, 결제 방식, 구독/렌탈 여부
6. 스펙 기반 필터링 (/quote-agent/hard-filters)
   → AI 추천 스펙 조건을 사용자가 커스터마이징
7. 예산 설정 (/quote-agent/budget)
8. AI 제품 추천 (/quote-agent/product-result, product-stream)
   → SSE 스트리밍으로 실시간 분석 결과
9. 제품 비교 (/quote-agent/product-compare)
   → 원본 견적 제품 vs AI 추천 대안 비교
10. 대리점 경쟁 입찰 → 최저가 견적
```

## 7.3 GABI 방식의 구조적 본질

**핵심 통찰**: GABI는 유저에게 **"무엇을 원하세요?"를 절대 묻지 않는다.**

대신:
- "가족이 몇 명이세요?"
- "얼마나 자주 요리하세요?"
- "주거 공간은 얼마나 되세요?"

이런 **답할 수 있는 질문**을 하고, 그 답으로 원하는 스펙을 **AI가 추론**한다.

### 유저 입장의 경험
- "자유롭게 입력하세요" → 뭘 입력해야 할지 모름 → 포기
- "질문에 답하세요" → 답할 수 있음 → 완주

### 시스템 입장의 경험
- 자유 입력 → LLM 매직 필요 → 12+ 모델 앙상블 필요
- 구조화 입력 → 결정적 로직 가능 → 단순 매칭으로 충분

## 7.4 송진우 피드백과의 합류

여기서 놀라운 일이 벌어졌다. GABI 방식이 **송진우가 직접 요청한 것과 정확히 일치**한다는 발견.

### 대조표

| 송진우의 요청 | GABI가 하는 것 |
|-------------|---------------|
| "자유도를 낮췄으면 좋겠다" | 단계별 명시적 선택 |
| "재검색 시 범주를 안 벗어나야" | 레퍼런스(견적서) 기준으로 모든 단계 진행 |
| "선택지가 많아지면 피로" | 각 단계가 검색 공간을 좁힘 |
| "같은 결의 대안" | 원본 vs 대안 명시적 비교 뷰 |

**→ 이건 우연이 아니다. 유저가 원하는 UX의 본질이 같다는 것.**

## 7.5 24명 인터뷰와의 합류

더 확장해서 보면, **24명 인터뷰 전원의 공통 Pain Point("취향 언어화 실패")에 대한 정확한 처방이 GABI 방식**이다.

| 인터뷰 인용 | GABI 방식의 대응 |
|-----------|----------------|
| 허자연: "프롬프트 쓰기도 어려울 수 있다" | 프롬프트 쓸 필요 없음, 질문에 답만 하면 됨 |
| 이장혁: "중간점이 없다는 느낌" | 스타일 감도 슬라이더로 "얼마나 비슷한가" 명시 |
| 김지혜: "내가 원하는 추구미대로 대분류를 먼저" | 속성 추출 후 유저가 핵심 속성 선택 |
| 여지혜: "네이밍을 모르는 사람들이 많음" | 단어 대신 시각적 옵션 제시 |

**→ 인터뷰가 말한 것, 송진우가 직접 요청한 것, GABI가 증명한 것이 모두 같은 방향을 가리킨다.**

## 7.6 Snapdeck 비유의 완성

이제서야 Snapdeck 비유가 완전히 이해된다.

### Snapdeck의 차별화
- 기존: **템플릿(미리 만든 구조)** + 텍스트 → 한정적
- Snapdeck: **LLM이 코드로 직접 그림** → 무제한

**제약을 우회하는 다른 메커니즘**: "템플릿에 채우기"가 아니라 "코드로 그리기".

### portal.ai의 차별화 (발견됨)
- 기존 (Daydream): **자유 프롬프트** + LLM 해석 → 희석, 피로, 좌절
- portal.ai: **체계적 Q&A** + 결정적 매칭 → 범주 유지, 명시적 선택, 만족

**제약을 우회하는 다른 메커니즘**: "자유 대화로 해석"이 아니라 "구조화된 답으로 추론".

### 본질적 전환
두 경우 모두:
- 기존은 **"한 번에 마법처럼 해결"**하려다 한계
- 새 방식은 **"과정을 쪼개서 각 단계에서 명확하게"** 처리

Snapdeck: 슬라이드 하나를 "한 번에 그리기" → "코드로 단계별로 그리기"
portal.ai: 검색을 "한 번에 매칭" → "Q&A로 단계별로 좁혀가기"

**→ 드디어 Daydream의 판을 빠져나올 수 있는 메커니즘이 보인다.**

# Part 8. 선택한 방향 — 체계적 Q&A 에이전트

## 8.1 최종 포지셔닝

> **자유 프롬프트 대신 체계적 Q&A로 유저의 니치한 취향을 단계별로 해체해, 범주가 희석되지 않는 대안을 찾아주는 패션 에이전트**

## 8.2 Daydream vs portal.ai 재정의

| 축 | Daydream | portal.ai |
|---|---------|-----------|
| **제품 철학** | "자유롭게 물어봐, LLM이 다 해줄게" | "체계적으로 물어볼게, 네가 답하면 정확히 찾아줄게" |
| **입력 방식** | 자연어 채팅 | 단계별 구조화 Q&A |
| **핵심 메커니즘** | 12+ 모델 앙상블 + Knowledge Graph | Q&A 기반 검색 공간 축소 + 결정적 매칭 |

## 8.4 재탐색 시 플로우 (송진우의 핵심 요구)

송진우의 핵심 요구: **"재검색 시 범주가 안 벗어나야"**.

재탐색 플로우 설계:
```

  ↓

  ↓

  ↓
결과: "범주가 유지된 채 색만 바뀐 대안"
```

**→ 기존 Q&A 답변이 state로 남아 있기 때문에, 자유 프롬프트와 달리 범주가 희석되지 않는다.**

# Part 9. 왜 이게 답인가 — 구조적 강점 분석

## 9.1 강점 1. 유저 Pain에 정확한 처방

| Pain | GABI 스타일 Q&A의 해결 |
|------|----------------------|
| 취향을 언어화 못 함 | 프롬프트 쓸 필요 없음, 질문에 답만 함 |
| "뭘 원하는지" 자체가 모호 | AI 파싱 결과를 보여주고 검증/수정 |
| 결과가 희석됨 | 핵심 속성 선택으로 명시적 locking |
| 선택지 과다로 피로 | 각 단계가 검색 공간을 좁힘 |
| 재탐색 시 범주 벗어남 | Q&A 답변이 state로 유지됨 |

**→ 인터뷰에서 발견된 모든 Pain에 정확히 대응.**

## 9.2 강점 2. 기술 비용 최소화

| 항목 | Daydream | portal.ai (Q&A) |
|------|----------|-----------------|
| LLM 모델 수 | 12+ | 1-2 (GPT-4o-mini 수준) |
| LLM 호출 | 모든 쿼리 | 파싱 단계만 |
| 검색 로직 | 복잡한 멀티모달 매칭 | 결정적 enum 매칭 |
| 전문가 팀 | 상시 필요 | Q&A 초기 설계만 |
| 운영 비용 | 높음 | 낮음 |

**핵심 원리**: 각 Q&A 단계가 검색 공간을 명시적으로 좁혀주므로, LLM 매직에 의존할 필요 없음.

## 9.3 강점 3. DB 커버리지 의존도 낮춤

현재 45K 상품에서 category 필드가 제대로 채워지지 않은 상태라도:
- 유저가 Step 2에서 속성을 검증했으므로 → 매칭할 속성이 명확
- 유저가 Step 3에서 핵심 속성을 선택했으므로 → 어느 속성이 strict match인지 명확
- DB에 해당 속성이 없어도 → 명시적으로 "매칭 불가"로 처리 가능 (블랙박스 아님)

**→ DB 품질 개선은 점진적으로 해도 제품은 작동함.**

## 9.4 강점 4. Daydream 정면 회피

이건 기술 경쟁이 아니라 **제품 철학의 차이**.

- Daydream이 전문가 100명을 더 채용해도 → 그들은 "자유 대화를 더 잘 해석"하는 쪽으로 개선할 것
- Daydream이 $100M을 더 유치해도 → 같은 판에서 더 크게 할 뿐

Q&A 방식은 Daydream의 **비즈니스 모델과 충돌**한다:
- Daydream은 "대화형 AI 쇼핑 어시스턴트"로 포지셔닝 → 투자자 스토리, PR 자산 전부 이 방향
- 갑자기 "체계적 Q&A"로 전환하면 기존 제품 철학 부정

**→ Daydream이 따라오려면 회사 전체 포지셔닝을 재설계해야 함. 쉽지 않음.**

## 9.5 강점 5. 2인 팀에 실행 가능

| 난이도 | 기술 경쟁 | Q&A 방식 |
|--------|----------|---------|
| 필요 인력 | 패션 전문가 + ML 엔지니어 + 데이터 엔지니어 | UX 디자이너 + 제품 엔지니어 |
| 1인당 생산성 | 낮음 (팀워크 필요) | 높음 (개인 작업 가능) |
| 반복 주기 | 느림 (모델 재학습 등) | 빠름 (Q&A 튜닝은 즉시) |
| 초기 비용 | 매우 큼 | 상대적으로 작음 |

2인 팀이 Q&A 기반 제품을 만드는 것은 **현실적으로 가능**하다.

## 9.6 추가 강점 — 확장 경로가 자연스러움

이 방향은 단기 차별화이면서도 장기 확장의 기반이 된다:

```
Phase 1: Q&A 기반 검색 (MVP)
  ↓
Phase 2: Q&A 결과 + 찜 누적 → 유저 취향 프로필 자동 학습
  ↓
Phase 3: 찜 기반 연쇄 추천 (Alta의 "보유 기반 추천"과 유사)
  ↓
Phase 4: 구매 이력 자동 등록 → 자연스러운 옷장 축적
  (의도적 디지타이징 없이)
  ↓
Phase 5: 맥락 기반 추천 (날씨, 일정)
  ↓
Phase 6: 핏/사이즈 개인화 (True Fit MCP 연동)
```

**→ 각 Phase가 이전 단계의 자연스러운 확장. 기능 폭주 리스크 없음.**

# Part 10. 리스크와 미해결 문제

이 방향이 정답이라도 실행 리스크는 존재한다. 정직하게 기록한다.

## 10.1 리스크 1. Q&A 길이가 드롭률을 높일 수 있다

### 위험
- GABI는 가전이라는 고관여 구매라서 긴 Q&A가 허용됨
- 패션은 더 가벼운 구매 → 단계가 길면 유저가 중간에 이탈

### 대응
- **단계 수를 최대 3-5개로 압축**
- 각 단계별 이탈률 측정 인프라 구축
- A/B 테스트로 최적 길이 탐색
- "Skip" 옵션을 일부 단계에 허용 (모르겠으면 건너뛰기)

### 미해결 질문
- 이상적 단계 수는 몇 개인가? (베타 테스트 필요)
- 어떤 단계가 가장 이탈률이 높을지?

## 10.2 리스크 2. 감성 매칭 vs 논리 매칭

### 위험
- 가전: "가족 수 → 냉장고 용량" 같은 rational 매핑이 가능
- 패션: "어떤 무드를 원하시나요?" 같은 질문은 여전히 유저가 답하기 어려움

### 대응
- **레퍼런스 제품을 항상 출발점**으로 두기 (0에서 시작하지 않음)
- 감성을 묻지 말고, **제품에서 추출된 속성을 보여주고 선택하게 하기**
- 예: "무드 뭘 원하세요?" (X) → "이 제품의 어떤 점이 좋으신가요?" (O)

### 미해결 질문
- 감성적 속성(무드, 톤앤매너)을 어떻게 구조화할 것인가?
- 스타일리스트 자문이 어디까지 필요한가?

## 10.3 리스크 3. "레퍼런스를 모르는 유저" 대응

### 위험
- 골든 유스케이스는 "발렌시아가 마이애미 더비"라는 구체적 레퍼런스
- 그러나 유저가 "그냥 예쁜 더비 신발 찾고 있어"라고 시작하면?

### 대응
- **Entry point를 3가지로 제공**:
  1. 구체적 레퍼런스 있음 (이미지/제품명/URL)
  2. 느낌만 있음 (텍스트 설명 → AI가 예시 이미지 생성/검색)
  3. 아무것도 없음 (카테고리부터 시작하는 탐색형 Q&A)

### 미해결 질문
- 어떤 비율의 유저가 구체 레퍼런스 vs 느낌만 갖는가?

## 10.4 리스크 4. Q&A 설계가 도메인 지식을 요구

### 검색 엔진 v5 설계

현재 v4 (enum 매칭 + gradient scoring)를 확장:

```python
def search_v5(qa_state):
    hard_filters = {
        'category': qa_state.category,
        'price_max': qa_state.budget,
        'platforms': qa_state.preferred_platforms,
    }

    # 핵심 속성 = strict match (가중치 1.0)
    strict_attributes = qa_state.locked_attributes

    # 비핵심 속성 = soft match
    # 가중치는 스타일 감도 슬라이더에 따라 조절
    soft_weight = compute_soft_weight(qa_state.style_tolerance)
    soft_attributes = qa_state.parsed_attributes - strict_attributes

    candidates = db.query(
        filters=hard_filters,
        must_match=strict_attributes,  # strict
        should_match=soft_attributes,  # soft
        weights={
            **{attr: 1.0 for attr in strict_attributes},
            **{attr: soft_weight for attr in soft_attributes},
        }
    )

    scored = score_candidates(candidates, qa_state)
    return rank_and_explain(scored)
```

## 11.4 LLM 사용 최소화

기존 구조에서 LLM 호출이 자주 필요한 부분 vs 새 구조:

| 단계 | 기존 (프롬프트 방식) | 새 구조 (Q&A 방식) |
|------|-----------------|------------------|

| 결과 설명 | LLM (매번) | LLM (결과 후 1회, 혹은 템플릿 기반) |

**→ LLM 호출 빈도 약 70% 감소 예상**. 운영 비용 직접 영향.

> "구체적인게 떠오르지 않은 상태에서 검색을 함. 번화가에서 입을 수 있는 그런 패션 이렇게 침. 단정한 것만 나올 때도 있고 유니크한 것만 나올 떄도 있고 해서 나의 취향과 안맞는다는 생각이 든다. 중간점이 없다는 느낌이 듬."

> "옷을 사고는싶은데 누가 맘에 쏙드는걸 찾아주면 좋겠다. 개인화가 잘 되어서 추천이 잘 되는 플랫폼이 있으면 좋겠다"
