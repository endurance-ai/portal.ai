💡 pgvector vs pgroonga — 둘의 역할

둘 다 검색에 쓰이지만 신호 종류가 다르다.

  • pgvector = 임베딩(벡터) 검색. 모델이 만든 768/1024차원 벡터를 HNSW 인덱스로 빠르게 nearest neighbor 조회. "의미적으로 비슷한 것" 을 찾는다 (예: 상품 이미지의 분위기·실루엣 유사도, 브랜드 컨셉 유사도).
  • pgroonga = 키워드(텍스트) 검색. 한글 형태소 분석 후 BM25 비슷한 sparse 매칭. "단어가 맞는 것" 을 찾는다 (예: 브랜드명 정확 매칭, "린넨"·"오버사이즈" 같은 단어 hit).

임베딩은 의미는 잘 잡지만 "이 브랜드명" 같은 정확 매칭이 약하고, 키워드는 "비슷한 느낌" 매칭이 안 됨 — 그래서 둘 다 필요. search_products_v5 가 두 신호를 RRF (Reciprocal Rank Fusion) 로 합쳐 hybrid 검색을 수행 (§7.1 참조). 한쪽 leg 만 강해도 보완되도록 설계.

💡 halfvec vs vector — 왜 상품은 halfvec(768), 브랜드는 vector(1024)?

pgvector 0.7+ 에 추가된 halfvec 은 float16 (반정밀도) 타입. 기본 vector 는 float32.

  • vector(768) = 4 bytes × 768 = 3 KB / row
  • halfvec(768) = 2 bytes × 768 = 1.5 KB / row — 50% 절약

HNSW 인덱스 사이즈도 거의 절반 (135 MB vs 270 MB 추정). L2-normalized 임베딩의 경우 float16 의 정밀도 손실은 cosine 유사도 결과에 실질적 영향 거의 없음.

  • products.embedding halfvec(768) — 33k → 81k+ 로 스케일 예정. 50% 절약 가치 큼. FashionSigLIP 가 L2-normalized output 이라 정밀도 손실 무리 없음.
  • brand_nodes.embedding vector(1024) — 수백 rows 라 스토리지 절약 의미 없음. BGE-m3 가 1024-dim full precision 으로 학습됐고 그대로 보존.

dense (HNSW <=>) + sparse (pgroonga &@~ + pgroonga_score) 두 leg 를 각자 k*4 뽑고, RRF (1/(rrf_k+rank)) fuse → LIMIT k. dense leg 는 embedding IS NOT NULL AND in_stock=true 강제, sparse leg 는 query_text 비면 skip. 호출처: kikoai/ai app/pipeline/search.py:54 (단일).

7.2 bulk_update_product_embeddings(jsonb) → int

payload = [{id, embedding, model}, ...] jsonb 배열 받아 UPDATE products SET embedding=halfvec(768), embedding_model, embedded_at=now() 일괄 실행. 호출: kikoai/ai/scripts/embed_batch_local.py:160 (FashionSigLIP 로컬 배치, 2026-05-10 확정 — Modal 미사용).

7.3 set_hnsw_ef_search(int) → void
🔴 DROP

세션 단위 hnsw.ef_search 튜닝 (recall vs latency). 호출처 0 — drop 안전.

7.4 get_product_filter_counts() → SETOF (dimension, value, count)

어드민 상품 필터 드롭다운 동적 옵션 집계. 7개 dimension (platform, category, style_node, color_family, fit, fabric, subcategory:<category>) UNION ALL. 호출: kikoai/app/src/app/api/admin/products/filter-options/route.ts:28 (10분 캐시 권장).

§9. 운영 메모

9.1 사용자 권한 + RLS

  • 17 테이블 모두 ENABLE ROW LEVEL SECURITY 박힘 — 정책은 Supabase 시절 auth.role()='authenticated' 등 의존
  • self-host cutover 시 4 user (app_user / ai_user / admin_user / backup_user) 모두 BYPASSRLS 부여 → 정책 우회, 코드 수정 없이 동작. 크롤러는 별도 user 없이 app_user 활용 (§1.3 일관)
  • 향후 자체 모델 재작성하려면 auth.uid() → current_setting('app.user_id') GUC 로 치환 필요. 현 시점 (P7 완료, 2026-05-12) BYPASSRLS 유지 — v6 새 설계 시 RLS 자체 재작성 여부 결정
